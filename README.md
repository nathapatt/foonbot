# Foon AQI Bot

**Foon AQI Bot** is a Spring Boot LINE bot for checking air quality, saving user-specific AQI history, sending scheduled notifications, and generating short health guidance from recent AQI history, designed with Thai users in mind.

## Features

- Real-time AQI fetch from IQAir
- LINE reply, push, and broadcast messaging flows
- Flex message response for AQI details
- LIFF page for checking AQI from the user's current location
- LIFF page for AQI history and trend
- LIFF page for daily notification settings
- Per-user AQI history linked to LINE user ID
- Daily scheduled notifications by user time and timezone
- Health-guideline chat reply generated from AQI history
- Groq integration with safe fallback to rule-based health text
- LINE ID token verification for LIFF-protected APIs
- LINE webhook signature verification
- Admin-secret protection for manual trigger endpoints

## Why Thailand

This project is aimed at Thai users, especially in places where PM2.5 and seasonal haze are a serious concern, such as Chiang Mai and other high-risk areas in Thailand.

Instead of asking users to install a separate app, Foon AQI Bot uses LINE because it is already one of the most familiar and widely used messaging platforms in Thailand. That makes AQI updates, alerts, and health guidance easier to access in a channel people already check every day.

## Project Sections

The LINE rich menu is designed around 4 user-facing sections:

1. `Check AQI Right Now`
   Opens LIFF and asks for location to fetch nearby AQI.
2. `History / Trend`
   Opens LIFF and shows recent AQI trend plus history list.
3. `AQI Schedule Settings`
   Opens LIFF and lets the user enable/disable daily notifications and choose time.
4. `Health Guideline`
   Sends a text command in LINE chat and returns a short health-guideline response based on recent AQI history.

## Setup

1. Clone this repository.
2. Create `.env` in the project root from `.env.example`.
3. Fill in your environment variables:

```env
IQAIR_API_KEY=your_iqair_api_key_here
LINE_CHANNEL_TOKEN=your_line_channel_token_here
LINE_CHANNEL_SECRET=your_line_channel_secret_here
LINE_LOGIN_CHANNEL_ID=your_line_login_channel_id_here
GROQ_API_KEY=your_groq_api_key_here
GROQ_MODEL=openai/gpt-oss-120b
LIFF_AQI_ID=your_aqi_liff_id_here
LIFF_SETTINGS_ID=your_settings_liff_id_here
LIFF_HISTORY_ID=your_history_liff_id_here
APP_ADMIN_SECRET=choose_a_long_random_admin_secret_here
POSTGRES_USER=your_postgres_user_here
POSTGRES_PASSWORD=your_postgres_password_here
POSTGRES_DB=your_postgres_db_here
```

Environment notes:

- `LINE_CHANNEL_TOKEN` comes from the Messaging API channel and is used when this app calls LINE reply, push, and broadcast APIs.
- `LINE_CHANNEL_SECRET` comes from the Messaging API channel `Basic settings` tab and is used to verify incoming webhook signatures.
- `LINE_LOGIN_CHANNEL_ID` comes from the LINE Login channel `Basic settings` tab and is used to verify LIFF ID tokens.
- `APP_ADMIN_SECRET` is generated by you. Use a long random value, for example `openssl rand -hex 32`.

4. Start PostgreSQL:

```bash
docker compose up -d
```

5. Run the app:

```bash
./mvnw spring-boot:run
```

## LINE Console Setup

### LIFF Apps in LINE Developers

Create 3 LIFF apps under your LINE Login channel and point each one to your app endpoint:

- AQI page
  - Endpoint URL: `https://<your-cloudflared-domain>/liff/aqi`
- History page
  - Endpoint URL: `https://<your-cloudflared-domain>/liff/history`
- Settings page
  - Endpoint URL: `https://<your-cloudflared-domain>/liff/settings`

Example during local development:

```text
https://wilderness-boards-valued-onion.trycloudflare.com/liff/aqi
```

Each LIFF app must include the `openid` scope so `liff.getIDToken()` can return an ID token. The `profile` scope is also recommended for normal LIFF profile usage.

Required LIFF scopes:

- `openid`
- `profile`

After creating the LIFF apps, copy each LIFF ID into `.env`:

```env
LIFF_AQI_ID=your_aqi_liff_id_here
LIFF_SETTINGS_ID=your_settings_liff_id_here
LIFF_HISTORY_ID=your_history_liff_id_here
```

Also copy the LINE Login channel ID into `.env`:

```env
LINE_LOGIN_CHANNEL_ID=your_line_login_channel_id_here
```

If your LIFF ID looks like `2001234567-AbCdEfGh`, the numeric prefix is usually the LINE Login channel ID, but confirm it in `Basic settings -> Channel ID`.

## Usage

### Local API

Base path: `/api/air-quality`

- `GET /notify` - admin-only endpoint that fetches AQI and broadcasts a LINE message
- `GET /fetch` - admin-only endpoint that fetches AQI and saves it without sending a LINE message
- `GET /latest` - get the latest AQI record
- `GET /history` - get the latest global AQI records
- `GET /history/me?limit=30` - get user-specific AQI history
- `POST /by-location` - fetch AQI from the user's location and push AQI result to that LINE user
  - body: `{ "lat": 13.75, "lon": 100.50 }`

Base path: `/api/users`

- `GET /me/settings` - get notification settings for the signed-in LINE user
- `PUT /me/settings` - update `notifyEnabled`, `notifyTime`, and `timezone`

Protected LIFF endpoints now require:

- `Authorization: Bearer <LINE_ID_TOKEN>`

Admin-only manual trigger endpoints require:

- `X-Admin-Secret: <APP_ADMIN_SECRET>`

Example admin request:

```bash
curl -H "X-Admin-Secret: $APP_ADMIN_SECRET" \
  http://localhost:8080/api/air-quality/fetch
```

Client requests must not send `userId`. For LIFF-protected endpoints, the backend verifies the LINE ID token and derives the LINE user ID from the verified token.

### LIFF Routes

- `/liff/aqi` - AQI check page
- `/liff/history` - AQI history and trend page
- `/liff/settings` - daily notification settings page

All 3 LIFF pages use IDs from environment-based config instead of hardcoded IDs.

User-facing rich menu links should open LIFF through LINE:

```text
https://liff.line.me/<LIFF_ID>
```

### LINE Chat Commands

Send one of these messages in LINE chat:

- `คำแนะนำสุขภาพ`
- `Health Guideline`
- `Health Guidelines`
  Replies with a short health-guideline text generated from the user's recent AQI history.

AQI checking for normal users happens through the LIFF AQI page at `/liff/aqi`, where the user shares live location.

### Scheduler Behavior

- The scheduler runs every minute.
- It checks each user's configured notification time and timezone.
- It sends only to users who have notifications enabled and are due at that minute.
- It requires saved location data, so the user must check AQI at least once before scheduled pushes can work.

### Health Guideline Flow

- The app loads recent AQI history for the current LINE user.
- It computes summary facts such as latest AQI, trend, min/max, average, and dominant pollutant.
- It sends that summary to Groq to generate a short Thai response.
- If Groq is unavailable, the app falls back to rule-based text.
- The LLM is used for natural-language guidance, not for medical diagnosis.

## LINE Webhook (Local)

1. Start the app on port `8080`.
2. Start Cloudflare Tunnel:

```bash
cloudflared tunnel --url http://localhost:8080
```

3. Set the LINE webhook URL in LINE Developers Console:

```text
https://<your-cloudflared-domain>/api/line/webhook
```

4. Add these values to `.env`:

```env
LINE_CHANNEL_TOKEN=your_line_channel_token_here
LINE_CHANNEL_SECRET=your_line_channel_secret_here
```

5. Ensure:

- Use webhook = ON
- Response mode = Bot
- Auto-response messages = OFF

6. Test in LINE chat with:

```text
คำแนะนำสุขภาพ
```

The webhook handler rejects requests that don't include a valid LINE `X-Line-Signature` header.

## Security Flow

### LIFF Requests

The app doesn't trust a `userId` sent from the browser. Instead:

1. The user opens a LIFF page through `https://liff.line.me/<LIFF_ID>`.
2. The LIFF SDK initializes and obtains a LINE ID token through the `openid` scope.
3. The LIFF page sends `Authorization: Bearer <LINE_ID_TOKEN>` to the backend.
4. The backend verifies the ID token with LINE using `LINE_LOGIN_CHANNEL_ID`.
5. The backend reads the verified token's `sub` value as the LINE user ID.

This protects user-specific routes from IDOR-style attacks where a caller tries to pass another user's `userId`.

### Webhook Requests

LINE signs every webhook request with `X-Line-Signature`. The app recalculates the signature from the raw request body and `LINE_CHANNEL_SECRET`.

- Valid signature: process the webhook event.
- Invalid signature: reject with `401`.
- Missing `LINE_CHANNEL_SECRET`: reject with `503`.

### Sensitive Data

- Exact saved coordinates aren't returned from the settings API.
- Public AQI DTOs don't expose LINE user IDs.
- Manual trigger endpoints require `X-Admin-Secret`.

## Development Notes

- Custom Spring properties are documented in `META-INF/additional-spring-configuration-metadata.json` for better IDE support.
- `application.properties` loads `.env` automatically during local development.
- Section 4 health guidance is a chat reply, not a LIFF page.
- User history is only available after records are associated with a LINE user.
- LIFF pages now rely on LINE ID token verification instead of trusting a browser-supplied `userId`.
- If `liff.getIDToken()` returns empty, confirm the LIFF app has the `openid` scope.
- Run tests with `./mvnw test`.

## Security Notes

- Never commit real API keys or LIFF IDs meant to stay private.
- Keep `.env` local only.
- Rotate secrets immediately if leaked.
- Use managed secrets and a stable public domain in production.
- Keep Messaging API and LINE Login channels under the same provider so LINE user IDs line up across LIFF and webhook/Messaging API flows.
